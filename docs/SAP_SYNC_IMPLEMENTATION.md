# SAP BusinessPartner Sync Implementation

## Overview

This document describes the implementation of Portal → SAP BusinessPartners synchronization, completing the data flow from Google Form → Portal → SAP.

**Implementation Date:** December 30, 2025

---

## What Was Implemented

### 1. SAP Service Methods (`lib/services/sapService.js`)

Added three new methods to the SAP service:

- **`createBusinessPartner(businessPartnerData, sessionCookies)`**
  - Creates a new BusinessPartner in SAP
  - Endpoint: `POST /b1s/v1/BusinessPartners`
  - Returns the created BusinessPartner data

- **`updateBusinessPartner(cardCode, businessPartnerData, sessionCookies)`**
  - Updates an existing BusinessPartner in SAP
  - Endpoint: `PATCH /b1s/v1/BusinessPartners('{CardCode}')`
  - Returns the updated BusinessPartner data

- **`businessPartnerExists(cardCode, sessionCookies)`**
  - Checks if a BusinessPartner exists by CardCode
  - Returns `true` if exists, `false` otherwise
  - Handles 404 errors gracefully

### 2. Data Transformation Utility (`lib/utils/sapBusinessPartnerTransform.js`)

Created a utility module for transforming Portal data to SAP format:

- **`transformToSAPBusinessPartner(customer, lead)`**
  - Transforms Portal customer and lead data to SAP BusinessPartner format
  - Handles address mapping (BPAddresses)
  - Handles contact employee mapping (ContactEmployees)
  - Splits full names into first/last name
  - Extracts zip codes from addresses
  - Validates CardCode format (alphanumeric, max 15 chars)

- **`validateBusinessPartnerData(businessPartnerData)`**
  - Validates BusinessPartner data before sending to SAP
  - Checks required fields (CardName)
  - Validates field lengths
  - Validates email format
  - Returns validation result with errors array

### 3. Sync API Endpoint (`pages/api/customers/sync-to-sap.js`)

Created a dedicated API endpoint for syncing customers to SAP:

- **Endpoint:** `POST /api/customers/sync-to-sap`
- **Request Body:**
  ```json
  {
    "customer_id": "uuid" // OR
    "customer_code": "CPXXXXX"
  }
  ```
- **Features:**
  - Accepts either `customer_id` or `customer_code`
  - Checks if customer already exists in SAP (prevents duplicates)
  - Fetches associated lead data for address/contact info
  - Transforms Portal data to SAP format
  - Validates data before sending
  - Creates BusinessPartner in SAP
  - Updates Portal customer_code if SAP generates different CardCode
  - Returns detailed response with action taken (created/existing)

Ghost portal addresses (FSM rows not in SAP) are removed on delta sync when SAP fetch is complete. Manual SAP re-key for quotations: see [SAP_ADDRESS_LOSS_INVESTIGATION.md](./SAP_ADDRESS_LOSS_INVESTIGATION.md).

### 4. Automatic Sync Integration (`pages/api/leads/[leadId]/create-job.js`)

Integrated SAP sync into the lead-to-customer conversion flow:

- **When:** Automatically triggered when a customer is created from a lead
- **Location:** After customer creation, before job creation
- **Behavior:**
  - Non-blocking: Job creation succeeds even if SAP sync fails
  - Checks for SAP session availability
  - Checks if customer already exists in SAP
  - Creates BusinessPartner if new
  - Logs success/warning messages
  - Updates customer_code if SAP generates different CardCode

## Data Flow

### Complete Flow: Google Form → Portal → SAP

```
1. User submits Google Form
   ↓
2. Manual sync triggered (POST /api/leads/sync)
   ↓
3. Lead created in Portal (leads table)
   ↓
4. User creates job from lead (POST /api/leads/:leadId/create-job)
   ↓
5. Customer created in Portal (customer table)
   ↓
6. [NEW] SAP sync automatically triggered
   ↓
7. BusinessPartner created in SAP
   ↓
8. Job created successfully
```

### Manual Sync Flow

```
1. User calls POST /api/customers/sync-to-sap
   ↓
2. System fetches customer from Portal
   ↓
3. System checks if exists in SAP
   ↓
4. If new: Creates BusinessPartner in SAP
   ↓
5. If exists: Returns existing BusinessPartner
   ↓
6. Updates Portal customer_code if needed
```

## Field Mappings

### Portal Customer → SAP BusinessPartner

| Portal Field | SAP Field | Notes |
| ------------ | --------- | ----- |
| `customer_code` | `CardCode` | Auto-generated by SAP if not provided |
| `customer_name` | `CardName` | Required, max 100 chars |
| `email` | `EmailAddress` | Optional |
| `phone_number` | `Phone1` | Optional |

### Portal Lead → SAP BPAddresses

| Portal Field | SAP Field | Notes |
| ------------ | --------- | ----- |
| `block` | `Block` | Optional |
| `unit` | `BuildingFloorRoom` | Optional |
| `address` | `Street` | Optional |
| N/A | `AddressType` | Default: `bo_ShipTo` |
| N/A | `Default` | Default: `tYES` |
| N/A | `Country` | Default: `SG` |
| N/A | `City` | Default: `Singapore` |

### Portal Lead → SAP ContactEmployees

| Portal Field | SAP Field | Notes |
| ------------ | --------- | ----- |
| `full_name` (split) | `FirstName` | Extracted from full_name |
| `full_name` (split) | `LastName` | Extracted from full_name |
| `handphone` | `Phone1` | Optional |
| `email` | `E_Mail` | Optional |
| N/A | `Name` | Default: `Contact1` |
| N/A | `Position` | Default: `null` |

## Error Handling

### Non-Blocking Sync

- SAP sync failures do not block job creation
- Errors are logged but don't throw exceptions
- User can manually sync later via API endpoint

### Duplicate Prevention

- Checks if BusinessPartner exists before creating
- Returns existing BusinessPartner if found
- Handles SAP duplicate errors gracefully

### Validation

- Validates data before sending to SAP
- Returns detailed validation errors
- Prevents invalid data from reaching SAP

## Usage Examples

### Automatic Sync (When Creating Job from Lead)

No action required - sync happens automatically when:
1. Creating a job from a lead
2. Customer is created in Portal
3. SAP session is available

### Manual Sync via API

```javascript
// Sync by customer_id
const response = await fetch('/api/customers/sync-to-sap', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  credentials: 'include', // Include SAP session cookies
  body: JSON.stringify({
    customer_id: 'uuid-here'
  })
});

// Sync by customer_code
const response = await fetch('/api/customers/sync-to-sap', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  credentials: 'include',
  body: JSON.stringify({
    customer_code: 'LEAD-EMAIL-123456'
  })
});
```

### Response Format

```json
{
  "success": true,
  "message": "Customer successfully synced to SAP",
  "businessPartner": {
    "CardCode": "LEAD-EMAIL-123456",
    "CardName": "John Doe",
    "EmailAddress": "john@example.com",
    "Phone1": "0943843938",
    "BPAddresses": [...],
    "ContactEmployees": [...]
  },
  "action": "created", // or "existing"
  "customer": {
    "id": "uuid",
    "customer_code": "LEAD-EMAIL-123456",
    "customer_name": "John Doe"
  }
}
```

## Notes

- **CardCode Auto-Generation:** SAP will auto-generate CardCode if not provided. The generated code may differ from Portal `customer_code`, so we update Portal after creation.

- **Session Requirements:** SAP sync requires valid SAP session cookies (`B1SESSION`, `ROUTEID`). If session is not available, sync is skipped (non-blocking).

- **Address Handling:** SAP uses separate `BPAddresses` collection. Portal stores address as text, which is parsed and mapped to SAP address structure.

- **Contact Employees:** SAP uses `ContactEmployees` collection. Portal stores contact info directly in customer/lead tables, which is mapped to SAP contact structure.
